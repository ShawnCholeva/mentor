# Mentor v3.2 — Accurate Data, Conversation Awareness, Growth Engine

## Problem Statement

Mentor v3.1 has three core issues discovered through dog-fooding:

1. **Broken data pipeline.** The `user-prompt-submit` hook writes only a cooldown timestamp to the session state file. The `stop-logger` expects coaching details (`coaching_triggered`, `type`, `message`, `tags`) but they're never there. Result: 54 of 55 interactions show `coaching_triggered: false` despite the debug log showing multiple interventions fired. The user model is evolving on garbage data.

2. **False positives on conversational follow-ups.** The evaluator sees each prompt in isolation. Normal mid-conversation prompts like "Tell me more about Option 1" or "yes do approach 2" get flagged as lacking context, even though Claude already has the full conversation. This fired 4 times in a single dog-fooding session. It makes the mentor feel dumb and teaches users to ignore it.

3. **No growth signal.** The interaction log captures only `intent` labels (vague/direct/unknown) and a broken `coaching_triggered` flag. The user model updater has almost nothing to reason about. After 55 interactions, the model has 0 strengths, 1 weakness, 1 intervention history entry. There's no mechanism to track whether a user incorporated feedback or improved over time.

## Success Criteria

- Coaching interventions are accurately logged in `interactions.jsonl`
- False positive rate on mid-conversation follow-ups drops to near zero
- User model has meaningful data (strengths, weaknesses, milestones) after 30 interactions
- At least 1 reinforcement fires per 10 coaching interventions
- User can run `/mentor model` and see concrete evidence of their growth

## Design

### 1. Fix the Session State Handoff

**Current behavior:** `user-prompt-submit` line 160 writes:
```json
{"last_intervention_ts": 1775357850}
```

**New behavior:** When coaching fires, write the full coaching state:
```json
{
  "last_intervention_ts": 1775357850,
  "coaching_triggered": true,
  "type": "correction",
  "message": "the coaching message...",
  "tags": ["vague-prompt"]
}
```

When coaching does NOT fire, write only:
```json
{"last_intervention_ts": 1775357850}
```

The `stop-logger` already reads these fields correctly — it just needs the data to be present.

**Changes in `hooks/user-prompt-submit`:**
- After the judgment is parsed (~line 154), write the full coaching state including `coaching_triggered: true`, `type`, `message`, and any `tags` to the session state file.
- Move the state write from before the coaching output (~line 160) to after the judgment is fully parsed, so the write has all the coaching data available.
- Early-exit paths that don't fire coaching continue to write the minimal cooldown-only state (or no state at all).

### 2. Conversation Awareness

**Pass turn_count to the evaluator.**

The hook already counts turns from the transcript. Add `turn_count` to the JSON payload sent to `evaluate-prompt.sh`.

**Changes in `hooks/user-prompt-submit`:**
- Compute TURN_COUNT from the transcript. Currently this is done in `stop-logger` (line 53-58) but not in `user-prompt-submit`. Add the same logic: `grep -c '"role":"user"'` on the transcript file. The transcript path is available via the hook's stdin input as `transcript_path`.
- Add `--argjson turn_count "$TURN_COUNT"` to the payload builder at ~line 137.

**Changes in `hooks/lib/evaluate-prompt.sh`:**
- Read `turn_count` from stdin payload (default 1).
- Add to the evaluator system prompt:

```
## Conversation Awareness
Turn count for this prompt: ${TURN_COUNT}

If turn_count > 1, the user is mid-conversation. Claude already has the full conversation context.
- References to prior discussion ("option 1", "the first one", "tell me more", "that approach", "both", "yes do X") are normal follow-ups. Do NOT flag these as lacking context.
- Short prompts (under 6 words) at turn > 1 are conversational, not vague. Raise your threshold — only intervene if the prompt is truly ambiguous even WITH conversation context.
- Only intervene on mid-conversation prompts for substantive issues: scope creep, missing diagnostics for debugging, fundamentally flawed approach. Context-reference is never a valid reason to intervene after turn 1.
```

### 3. Richer Interaction Logging

**Add prompt_summary and coaching_summary to log entries.**

**Changes in `hooks/stop-logger`:**
- Add `prompt_summary` field: the user's last prompt truncated to 200 characters. `LAST_PROMPT` is already extracted from the transcript for intent classification (~line 65). Truncate and include it in the log entry.
- Add `coaching_summary` field: when `coaching_triggered` is true, include the intervention type and a truncated version of the coaching message (first 150 chars). This comes from the session state file (available once section 1 is fixed).
- Updated log entry schema:

```json
{
  "id": "uuid",
  "session_id": "...",
  "timestamp": 1775357850,
  "intent": "direct",
  "skill_used": null,
  "turn_count": 8,
  "prompt_summary": "Tell me more about Option 1 and 2",
  "coaching_triggered": true,
  "intervention_type": "correction",
  "coaching_summary": "correction: Prompt references options without context...",
  "tags": []
}
```

### 4. Growth Engine

#### 4a. Before/After Tracking

When coaching fires on prompt N, store the coached prompt in the session state file under `last_coached_prompt`. On the *next* prompt (N+1) in the same session, the hook checks: does `last_coached_prompt` exist in the state? If so, include it in the evaluator payload along with a `previous_was_coached: true` flag.

The evaluator can then assess whether the user incorporated the feedback. This is not a separate evaluation call — it's additional context on the regular evaluation. If the evaluator sees improvement, it can fire a `reinforcement` intervention.

**Changes in `hooks/user-prompt-submit`:**
- When coaching fires: add `last_coached_prompt` (truncated to 300 chars) to the session state file alongside the other coaching fields.
- On every prompt: read the session state file before evaluation. If `last_coached_prompt` exists, include it and `previous_was_coached: true` in the evaluator payload. Then clear `last_coached_prompt` from the state (so it only applies to the immediately-following prompt).
- Important: the state file is currently deleted by `stop-logger` after reading (`rm -f "$STATE_FILE"` at line 50). The state file write/read lifecycle needs adjustment:
  - `user-prompt-submit` writes state (coaching data + last_coached_prompt)
  - `stop-logger` reads and logs the coaching data, but does NOT delete the file
  - `user-prompt-submit` on next prompt reads `last_coached_prompt`, then overwrites with new state
  - State files are already cleaned up by the orphan sweep for crashed sessions

**Changes in `hooks/lib/evaluate-prompt.sh`:**
- Read `previous_was_coached` and `last_coached_prompt` from payload.
- Add to system prompt when `previous_was_coached` is true:

```
## Previous Prompt Was Coached
The user's previous prompt received this coaching intervention. Assess whether their current prompt shows improvement. If it does, consider a "reinforcement" intervention — genuine, specific praise for what improved and why it matters.

Previous coached prompt: ${LAST_COACHED_PROMPT}
```

#### 4b. Growth Milestones in the User Model

Extend the user model schema with a `milestones` array:

```json
{
  "strengths": ["includes output format in complex prompts", "provides error context when debugging"],
  "weaknesses": ["short follow-up prompts when switching topics"],
  "current_focus": "Adding constraints and scope to feature requests",
  "recent_progress": "Vague prompt rate dropped — 2 of last 10 flagged vs 5 of first 10",
  "intervention_history": ["correction: vague prompt", "nudge: missing format", "reinforcement: improved specificity"],
  "milestones": [
    {"date": "2026-04-01", "observation": "Started including output format consistently"},
    {"date": "2026-04-04", "observation": "Vague prompt rate dropped from 50% to 20%"}
  ]
}
```

**Changes in `hooks/lib/update-user-model.sh`:**
- Update the system prompt to instruct the model to look for and record milestone moments. Add to rules:

```
9. milestones: concrete, dated observations of behavioral change. Add a milestone when you see a clear shift — not incremental fluctuation but a genuine pattern change. Keep only the last 10 milestones. Each should name the specific behavior that changed and evidence for the change.
10. When interaction logs include prompt_summary fields, use the actual prompt content to assess quality trends — not just the intent labels.
```

- Update the jq validation to accept the `milestones` field:

```jq
milestones: ((.milestones // [])[-10:])
```

#### 4c. Reinforcement from Growth Data

No additional file changes needed — this emerges naturally from the pieces above:
- The evaluator already supports the `reinforcement` intervention type
- With `previous_was_coached` data (4a), it can fire reinforcement on immediate improvement
- With the user model containing milestones and progress data, it can fire reinforcement on long-arc growth
- The system prompt already says "roughly 1 in 10 interventions should be reinforcement" — now it has the data to actually do it

## Files Changed

| File | What changes |
|---|---|
| `hooks/user-prompt-submit` | Full coaching state in session file; turn_count + before/after fields in evaluator payload; state file lifecycle (no longer deleted by stop-logger) |
| `hooks/stop-logger` | Add prompt_summary + coaching_summary to log entries; stop deleting session state file (let user-prompt-submit manage lifecycle) |
| `hooks/lib/evaluate-prompt.sh` | Read turn_count / previous_was_coached / last_coached_prompt; conversation-awareness rules; before/after reinforcement rules |
| `hooks/lib/update-user-model.sh` | Milestones schema; growth-aware system prompt rules; updated jq validation |

No new files. No new dependencies.

## What This Does NOT Change

- The coaching message format and injection mechanism (additionalContext) — it's working well
- The philosophy system — still user-editable, still feeds the evaluator
- The `/mentor` and `/mentor-recap` skills — they read the same files, just get richer data
- The cooldown system — still per-session, still configurable
- The model selection (Haiku for chill, Sonnet for elite) — unchanged

## Risk / Edge Cases

- **State file lifecycle change:** `stop-logger` currently deletes the state file after reading. New lifecycle: (1) `user-prompt-submit` reads existing state (for `last_coached_prompt`), (2) `user-prompt-submit` overwrites with new state, (3) `stop-logger` reads state but does NOT delete. Each prompt overwrites the previous state, so files don't accumulate — there's exactly one per active session. The existing orphan sweep (`find -mtime +1 -delete`) handles crashed sessions.
- **Evaluator prompt length:** Adding turn_count, conversation-awareness rules, and before/after context increases the system prompt size. Stay within reason — the evaluator call is latency-sensitive (25s timeout).
- **Milestone accuracy:** The user model updater (Haiku) may produce low-quality milestones initially. The "3+ instances before adding strengths" rule and conservative update instructions should prevent hallucinated milestones, but monitor the first few updates.
