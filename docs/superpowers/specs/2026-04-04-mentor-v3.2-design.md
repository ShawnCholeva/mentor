# Mentor v3.2 — Accurate Data + Conversation Awareness

## Problem Statement

Mentor v3.1 has three core issues discovered through dog-fooding:

1. **Broken data pipeline.** The `user-prompt-submit` hook writes only a cooldown timestamp to the session state file. The `stop-logger` expects coaching details (`coaching_triggered`, `type`, `message`) but they're never there. Result: 54 of 55 interactions show `coaching_triggered: false` despite the debug log showing multiple interventions fired. The user model is evolving on garbage data.

2. **False positives on conversational follow-ups.** The evaluator sees each prompt in isolation. Normal mid-conversation prompts like "Tell me more about Option 1" or "yes do approach 2" get flagged as lacking context, even though Claude already has the full conversation. This fired 4 times in a single dog-fooding session. It makes the mentor feel dumb and teaches users to ignore it.

3. **No growth signal.** The interaction log captures only `intent` labels (vague/direct/unknown) and a broken `coaching_triggered` flag. The user model updater has almost nothing to reason about. After 55 interactions, the model has 0 strengths, 1 weakness, 1 intervention history entry.

## Phasing

This work is split into two phases:

- **Phase 1 (this spec):** Fix the data pipeline, add conversation awareness, enrich interaction logging. Ship it and collect real data.
- **Phase 2 (future spec, after 1-2 weeks of real data):** Growth engine — before/after tracking, milestones, reinforcement from growth data. Requires Phase 1's accurate data to be meaningful.

This spec covers Phase 1 only.

## Success Criteria

- Coaching interventions are accurately logged in `interactions.jsonl` (coaching_triggered, intervention_type, message all present when coaching fires)
- False positive rate on mid-conversation follow-ups drops to near zero
- User model begins evolving meaningfully within 30 interactions of accurate data
- Intent classification: "unknown" is no longer the majority category

## Design

### 1. Fix the Session State Handoff

**Current behavior:** `user-prompt-submit` line 160 writes:
```json
{"last_intervention_ts": 1775357850}
```

**New behavior:** When coaching fires, write the full coaching state:
```json
{
  "last_intervention_ts": 1775357850,
  "coaching_triggered": true,
  "type": "correction",
  "message": "the coaching message..."
}
```

Note: the `tags` field is dropped from the schema. The evaluator never produces tags — the field was always empty. Remove it from the state file, the log entry, and `stop-logger`'s state file reader.

When coaching does NOT fire, write only:
```json
{"last_intervention_ts": 1775357850}
```

The `stop-logger` already reads `coaching_triggered`, `type`, and `message` from the state file — it just needs the data to be present.

**Changes in `hooks/user-prompt-submit`:**
- After the judgment is parsed and intervention type/message are extracted (~line 154), write the full coaching state to the session state file. Currently the state write happens at ~line 160 before the coaching output block. Move it to after the judgment is parsed so it has access to `INTERVENTION_TYPE` and `FEEDBACK`.
- The state write should use `jq` to build valid JSON with `coaching_triggered` as a JSON boolean (not a string).
- Early-exit paths that don't fire coaching continue to write the minimal cooldown-only state (or no state).

**Changes in `hooks/stop-logger`:**
- Remove the `TAGS` variable and `.tags` reader from the state file parsing (lines 47, 109).
- Remove `tags` from the log entry jq builder.
- Everything else in the state file reader already works — it just needs data to be present.

### 2. Conversation Awareness

**Derive turn count from the history array, not the transcript.**

The `UserPromptSubmit` hook fires *before* the prompt reaches Claude — `transcript_path` may not be available at this point. But the hook already reads prompt history from `~/.claude/history.jsonl` into `HISTORY_JSON` (lines 103-117). The length of this array is the turn count for the current session.

Even simpler: the evaluator already receives the `history` array in its payload. It can derive turn count itself by counting the array entries. This requires zero changes to `user-prompt-submit`.

**Changes in `hooks/lib/evaluate-prompt.sh`:**
- Read the `history` array from the payload (already done at line 23).
- Compute turn count from the history array length.
- Add `TURN_COUNT` to the system prompt with conversation-awareness rules:

```
## Conversation Awareness
This is turn ${TURN_COUNT} of the conversation (based on ${TURN_COUNT} previous prompts in session history).

If turn count > 1, the user is mid-conversation. Claude already has the full conversation context.
- References to prior discussion ("option 1", "the first one", "tell me more", "that approach", "both", "yes do X") are normal follow-ups. Do NOT flag these as lacking context.
- Short prompts (under 6 words) at turn > 1 are conversational, not vague. Raise your threshold — only intervene if the prompt is truly ambiguous even WITH conversation context.
- Only intervene on mid-conversation prompts for substantive issues: scope creep, missing diagnostics for debugging, fundamentally flawed approach. Context-reference is never a valid reason to intervene after turn 1.
```

### 3. Richer Interaction Logging

**Add `prompt_summary` to log entries.**

**Changes in `hooks/stop-logger`:**
- Add `prompt_summary` field: the user's last prompt truncated to 200 characters. `LAST_PROMPT` is already extracted from the transcript for intent classification (~line 65). Truncate and include it in the log entry.
- The `coaching_summary` field proposed in the original design is dropped — it's redundant with `intervention_type` + the message already available in the state file. The user model updater can assemble what it needs from `intervention_type` and `COACHING_MESSAGE`.
- Pass the coaching message through to the user model updater payload (it already does this when `COACHING_TRIGGERED` is true — this just starts working once Section 1 fixes the data).

Updated log entry schema:

```json
{
  "id": "uuid",
  "session_id": "...",
  "timestamp": 1775357850,
  "intent": "direct",
  "skill_used": null,
  "turn_count": 8,
  "prompt_summary": "Tell me more about Option 1 and 2",
  "coaching_triggered": true,
  "intervention_type": "correction"
}
```

Fields removed from schema: `tags` (evaluator never produces them).
Fields added: `prompt_summary`.

**Intent classification improvement:** The current heuristic (< 6 words = "vague", >= 6 words = "direct", else "unknown") produces "unknown" as the majority. Tighten this: any prompt with extractable text that isn't a skill invocation should be classified as either "vague" (< 6 words) or "direct" (>= 6 words). "unknown" should only occur when the prompt text can't be extracted from the transcript. This is a logic fix in the existing intent classification block (~lines 73-89).

### State File Lifecycle

**Current lifecycle:**
1. `user-prompt-submit` writes `{last_intervention_ts: ...}` to state file
2. `stop-logger` reads state file, then deletes it (`rm -f`)

**New lifecycle:**
1. `user-prompt-submit` writes full coaching state (or cooldown-only) to state file
2. `stop-logger` reads state file, does NOT delete it
3. Next `user-prompt-submit` overwrites with new state

Each prompt overwrites the previous state, so at most one state file exists per active session. The existing orphan sweep (`find -mtime +1 -delete`) handles crashed sessions.

**Known imprecision:** `stop-logger` runs asynchronously. If the user submits a new prompt before `stop-logger` finishes processing the previous one, `stop-logger` may read the *new* prompt's state instead of the old one. This is acceptable — interaction logging doesn't need perfect accuracy, and the cooldown (30s default) makes this race unlikely in practice. The coaching data from the missed prompt is still visible in the debug log.

## Files Changed

| File | What changes |
|---|---|
| `hooks/user-prompt-submit` | Write full coaching state (triggered, type, message) to session state file when coaching fires |
| `hooks/stop-logger` | Add `prompt_summary` to log entries; remove `tags` field; stop deleting session state file; improve intent classification to reduce "unknown" |
| `hooks/lib/evaluate-prompt.sh` | Derive turn count from history array; add conversation-awareness rules to system prompt |

No changes to `hooks/lib/update-user-model.sh` in Phase 1. It already works — it just needs accurate data from the fixed pipeline.

No new files. No new dependencies.

## What This Does NOT Change

- The coaching message format and injection mechanism (additionalContext) — it's working well
- The philosophy system — still user-editable, still feeds the evaluator
- The `/mentor` and `/mentor-recap` skills — they read the same files, just get richer data
- The cooldown system — still per-session, still configurable
- The model selection (Haiku for chill, Sonnet for elite) — unchanged
- The user model schema — unchanged in Phase 1 (milestones deferred to Phase 2)
- The user model updater (`update-user-model.sh`) — unchanged; it starts working better automatically once it receives accurate interaction data

## Phase 2 Preview (not in scope)

After 1-2 weeks of accurate data from Phase 1:

- **Before/after tracking:** When coaching fires, track whether the user's next prompt improves. Let the model updater detect this from prompt_summary data in the log, rather than wiring it through the evaluator.
- **Growth milestones:** Add a `milestones` array to the user model schema — concrete, dated observations of behavioral change.
- **Reinforcement from growth data:** Use milestones and accurate coaching history to trigger proactive positive reinforcement when the user improves in a previously-weak area.

These features need real data to be meaningful. Building them on the current broken pipeline would produce garbage milestones and false reinforcement.
